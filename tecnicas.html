Pruebas de software: tipos, niveles y técnicas.

Tipos

Funcionales

Evalúan qué hace el sistema, según los requisitos funcionales.

  • Caja Negra: Se prueban las funcionalidades sin conocer el código interno. Se enfoca en las entradas y salidas del sistema.
  • Aceptación: Pruebas realizadas por el cliente o usuario final para verificar si el sistema cumple con lo acordado (requisitos del negocio).
  • Regresión: Se asegura que las funcionalidades que ya funcionaban siguen haciéndolo después de cambios, correcciones o nuevas funcionalidades.
  • Integración (funcional): Prueban la interacción entre dos o más módulos para verificar que trabajen correctamente juntos.

No Funcionales

Evalúan cómo lo hace el sistema, no qué hace.

  • Rendimiento (Performance): Evalúa la rapidez, capacidad de respuesta y estabilidad del sistema bajo ciertas condiciones de carga.
  • Seguridad: Evalúa si el sistema protege los datos y mantiene la confidencialidad, integridad y disponibilidad.
  • Usabilidad: Verifica qué tan fácil e intuitivo es para los usuarios interactuar con el sistema.
  • Compatibilidad: Evalúa si el sistema funciona correctamente en diferentes entornos (navegadores, dispositivos, sistemas operativos, resoluciones, etc.).

Niveles de Pruebas

Refieren a cuándo y sobre qué parte del software se hacen las pruebas.

  • Pruebas Unitarias: Se prueban unidades pequeñas e individuales de código, como funciones o métodos. Normalmente las realiza el desarrollador.
  • Pruebas de Integración: Se verifica que múltiples módulos o componentes funcionen correctamente cuando se integran entre sí.
  • Pruebas de Sistema: Se prueba el sistema completo, como un todo, para asegurarse de que cumple con los requisitos funcionales y no funcionales.
  • Pruebas de Aceptación: Se realizan antes de liberar el producto. Las hace el cliente o usuario para validar que el software está listo para producción.
  • Caja Negra: Pruebas sin conocimiento del código interno. Se enfoca en entradas y salidas.
  • Partición de Equivalencia: Divide las entradas en grupos donde se espera el mismo comportamiento para reducir la cantidad de casos de prueba.
  • Análisis de Valores Límite: : Se prueban los valores en los extremos del rango permitido, donde suelen aparecer errores (por ejemplo, si una edad válida es de 18 a 60, se prueban 17, 18, 60, 61).
  • Caja Blanca: Pruebas con acceso al código. Se enfoca en la lógica interna.
  • Cobertura de Código: Mide cuánto del código fue ejecutado durante las pruebas (líneas, ramas, condiciones).
  • Análisis de Caminos: Analiza todos los caminos posibles que puede tomar la ejecución del código para cubrir todos los flujos lógicos.
  • Caja Gris: Pruebas con conocimiento parcial del código. Mezcla técnicas de caja blanca y negra.
  • Pruebas de Interfaz: Verifican la correcta comunicación entre distintos módulos o entre el sistema y componentes externos.
  • Pruebas de API: Evalúan si las interfaces de programación (REST, SOAP, etc.) funcionan como se espera al enviar y recibir datos.
